blooming_flower

diagnostic commands

(Extracted from loop()'s switch on characters read from serial)
'p' turn motor power on
'o' turn motor power off
' ' (space char) show sensor and motion state info
'.' (period) ask motor to stop moving
'/' act like sensor just became dry
',' act like sensor just became wet
'!' executes most of startup.
Manage configuration state:
'?' show configuration info (stuff that gets saved)
'\' (backslash) reset all to compiled in values (a bunch of assignment statements in class Puzzle)
'`' (back tick) Undoing temporary changes (reread EEProm)
'|' (vertical bar) Saving parameters (save into EEProm)

Tweakable parameters are selected by the following letters, w is the powerup default:
'w' wetness threshold WetnessRequired;
't' track length trackLength;
'h' wetness hysteresis WetnessHysteresis;
'a' acceleration Acceleration;
's' max speed MaxSpeed;

Values are raised or lowered by the letters used for Roman numerals, lower case is subtracted, upper case added: (function romanize)
  'i':1; 'v':5; 'x':10; 'l':50; 'c':100;  'd':500;  'm':1000.
For instance, to alter track length from 100 to 127:  "txxvii?".
You don't need to send them in one Serial Monitor click, characters are inspected individually andnewlines are ignored.
When you are happy with the config '|' saves it, and you can test it mostly with '!', but should test with a power cycle of the board before trusting it fully.

Reverse engineering

The internal control board has the motor connector,
a connector for an optical pickup for the home sensor,
a PWM output for the LED,
a UART connector,
4 stake pins connector that I either didn't figure out or forgot what it does,
and a connector for what is likely a MCU downloader/debugger.

I gather that to get the thing to do something more than its back and forth we need to talk to it via the serial port, then use the stake pins for triggers. Unfortunately the connector is metric spaced and I don't have the associated parts to make a nice cable.

Present state of the gear

The motor board is up on wings that make it easy to get to pins, they are however a bit crappy and we should remove them when not actively debugging the hardware.
The particular motor board that is mounted is one of mine, the original one with a long cable to the sensor is in the tote and it would be simplest to swap them out rather than move the wires over to my board.

The program thinks it is driving the flower open and closed for a fixed number of steps given by "track length". If we set the acceleration to the same or greater than the speed (named "MaxSpeed" for reasons) then we can use the MaxSpeed and the number of steps to compute the runtime, or actually the other way around, choose a number of steps (since we aren't doing the stepping choose any number) and then set the MaxSpeed to get the time that we want the thing to spend in each traversal.

The dangly bits that are presently not actually used

I have made a breadboard with connectors for the motor and the home sensor, and have made but need to solder on a T-tap for the home sensor, which beware uses 12V.

The home sensor data line is 12V (the associated power pin) through most of the travel, and ~0V when at one of the extremes, I did not note which. We can feed this through an optocoupler before passing on to the Arduino, or a voltage divider if we are feeling brave. I have a board receiving the signal which needs a little rework for this isolation/voltage conversion.
